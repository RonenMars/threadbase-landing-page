# Multi-language Support (Russian, Hebrew, Arabic) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Russian (`ru`), Hebrew (`he`, RTL), and Arabic (`ar`, RTL) locales to the Threadbase landing site using `next-intl`, with English (`en`) remaining the unprefixed default.

**Architecture:** Move every route under an `app/[locale]/...` dynamic segment. Extract all marketing copy out of `lib/content.ts` into four per-locale JSON message catalogs (`messages/en.json`, `messages/ru.json`, `messages/he.json`, `messages/ar.json`). Components read strings via `useTranslations`/`getTranslations` instead of importing constants. `next-intl`'s middleware handles locale detection (`Accept-Language` header + `NEXT_LOCALE` cookie), and a new `LanguageSwitcher` component lets visitors override it. Hebrew/Arabic get `dir="rtl"` on `<html>`, with global-chrome components (`NavMenu`, `Footer`, `PageShell`, `SectionHeading`, `BulletList`) converted from physical to logical Tailwind utilities so layout mirrors automatically.

**Tech Stack:** Next.js 16 App Router, React 19, TypeScript, Tailwind v4 (config in `app/globals.css`, no `tailwind.config.ts`), `next-intl` (new dependency), Vitest, Playwright.

## Global Constraints

- Path alias `@/*` maps to the project root, not `src/`.
- No Prettier — ESLint handles all formatting including Tailwind class order; run `eslint . --fix` after each task that touches JSX/CSS classes.
- No `tailwind.config.ts` — any new Tailwind theme values go in `app/globals.css`.
- Playwright visual tests run at 0.2% pixel tolerance — do not change parallelism or tolerance defaults; update snapshots only with `npm run test:visual:update` after confirming a visual diff is intentional.
- All changes go through a feature branch and PR — never commit directly to `main`. Branch name: `feat/i18n-ru-he-ar`.
- Commit messages: conventional-commit format (`<type>(<scope>)?: <description>`), no AI attribution, one sentence per line in any GitHub-bound text.
- Locale set is fixed at exactly 4: `en` (default, unprefixed), `ru`, `he` (RTL), `ar` (RTL). `localePrefix: "as-needed"`.
- Switcher displays ISO codes only (`EN` / `RU` / `HE` / `AR`), placed in both `NavMenu` and `Footer`.
- Terminal command lines inside `QUICK_START.steps` (e.g. `npm install -g @threadbase-sh/streamer`, `tb-streamer serve`) stay verbatim/untranslated in every locale; only `#` comment lines and prose translate.
- `Hero/DeviceMesh/*`, `Hero/GlitchTitle`, and `usePtyTypewriter` do not mirror under RTL — they keep fixed LTR visual orientation regardless of locale.
- Email addresses (`support@threadbase.sh`), URLs, and the `EFFECTIVE_DATE`/version strings never get translated.
- The working tree currently has `app/privacy/page.tsx` already moved to `app/privacy-policy/page.tsx` by in-progress uncommitted work on branch `fix/privacy-policy-slug` (not part of this feature). This plan's Task 1 branches from `main`, so **before starting Task 2 (content extraction), re-check whether `fix/privacy-policy-slug` has merged** — if it has, the privacy route is `app/[locale]/privacy-policy/page.tsx`; if it hasn't, use `app/[locale]/privacy/page.tsx`. This plan assumes the **not-yet-merged** state and uses `app/privacy/page.tsx` as the source path, since that's what exists on `main` today.

## File Structure

**New files:**
- `proxy.ts` — next-intl locale-detection middleware (named `middleware.ts` before Next.js 16)
- `i18n/routing.ts` — locale list, default locale, `localePrefix` config, typed `Link`/`redirect`/`usePathname`/`useRouter` helpers
- `i18n/request.ts` — next-intl server config (`getRequestConfig`) that loads the right message catalog per request
- `messages/en.json`, `messages/ru.json`, `messages/he.json`, `messages/ar.json` — per-locale message catalogs
- `components/LanguageSwitcher.tsx` — new switcher component (client), used in both `NavMenu` and `Footer`
- `lib/content.ts` — kept, but trimmed to only non-string structural config (icon names, hrefs, CTA variants, roadmap `status` keys, platform `id`s) zipped with translated strings by index at render time
