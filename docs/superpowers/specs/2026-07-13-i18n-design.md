# Multi-language support (Russian, Hebrew, Arabic)

## Goal

Add Russian, Hebrew, and Arabic translations to the Threadbase landing site, with Hebrew and Arabic rendering right-to-left. English remains the default/fallback locale.

## Library

`next-intl` — purpose-built for Next.js App Router: server-component-friendly message loading, built-in routing/middleware, and locale-prefixed static generation.

The mobile app (`tb-mobile`) uses `i18next`/`react-i18next`, but its message catalogs (`locales/en/*.json`) are app-UI chrome — buttons, session/terminal strings, onboarding flow — with no content overlap with the landing page's marketing copy. No string reuse is possible; only terminology consistency is borrowed (see "Translation content" below).

## Routing

- Routes move under a `[locale]` dynamic segment: `app/[locale]/page.tsx`, `app/[locale]/betas/page.tsx`, `app/[locale]/solutions/page.tsx`, `app/[locale]/support/page.tsx`, `app/[locale]/android-beta/page.tsx`, `app/[locale]/privacy/page.tsx`.
- Locales: `en` (default, unprefixed), `ru`, `he`, `ar`.
- `localePrefix: "as-needed"` — English serves at `/`, `/betas`, etc. Others serve at `/ru`, `/ru/betas`, `/he`, `/ar/support`, etc.
- `proxy.ts` (next-intl's `createMiddleware`; the file was named `middleware.ts` before Next.js 16) handles locale negotiation:
  - First visit: read `Accept-Language`, redirect to the best-matching locale if not English.
  - Store the resolved locale in a `NEXT_LOCALE` cookie so subsequent visits don't get re-redirected.
  - A manual switcher (see below) always wins over both header and cookie once clicked.

## Language switcher

- New small client component `components/LanguageSwitcher.tsx`.
- Displayed in two places: inside `NavMenu`'s mobile panel (near the bottom, above the "Threadbase v1.0" footer block) and in `Footer`, next to the existing link list.
- Displays ISO codes: `EN`, `RU`, `HE`, `AR` as plain text links/buttons.
- Clicking swaps only the locale segment of the current path (preserves the current page — e.g. on `/betas` switching to Russian goes to `/ru/betas`) and sets the `NEXT_LOCALE` cookie.

## Content restructuring

- `lib/content.ts`'s exported constants are removed; their string values move into per-locale JSON message catalogs: `messages/en.json`, `messages/ru.json`, `messages/he.json`, `messages/ar.json`.
- Namespacing mirrors the existing sections in `lib/content.ts`: `hero`, `problem`, `howItWorks`, `features`, `honestCons`, `roadmap`, `quickStart`, `footer`, `nav`, `betas`, `androidBeta`, `support`, `solutions`, `privacy`.
- Non-string structural data that isn't translatable (icon names, hrefs, CTA `variant`, roadmap `status`) stays in TypeScript alongside the components (or a slim `lib/content.ts` retained just for structural/non-string config), keyed so it can be zipped with the translated strings by key/index at render time.
- Components switch from importing constants directly to `useTranslations("section")` (client components) or `getTranslations({ locale, namespace: "section" })` (server components/`generateMetadata`).
- `QUICK_START.steps` (the terminal command block) is translated line-by-line, but shell commands themselves (`npm install -g ...`, `tb-streamer serve`, etc.) stay verbatim in every locale — only the `# comment` lines and step-4 prose get translated.

## RTL

- Locale layout (`app/[locale]/layout.tsx`) sets `<html dir={locale === "he" || locale === "ar" ? "rtl" : "ltr"} lang={locale}>`.
- Global-chrome components (`NavMenu`, `Footer`, `PageShell`, `SectionHeading`, `BulletList`, `FloatingDock`) get their hardcoded physical Tailwind utilities (`ml-`/`mr-`/`pl-`/`pr-`/`left-`/`right-`/`text-left`/`text-right`) converted to logical equivalents (`ms-`/`me-`/`ps-`/`pe-`/`start-`/`end-`/`text-start`/`text-end`) so layout mirrors automatically under `dir="rtl"`.
- Decorative/visual pieces that should **not** mirror: `Hero/DeviceMesh/*` (3D laptop/phone/server scene), `Hero/GlitchTitle`, and the terminal typewriter effect (`usePtyTypewriter`) — these keep their fixed LTR visual orientation regardless of locale, since mirroring a 3D scene or terminal output would look broken rather than correct.

## Translations

- I write the Russian, Hebrew, and Arabic copy directly for every page in scope: home, `/betas`, `/solutions`, `/support`, `/android-beta`, `/privacy`, nav, footer. Full parity across all four locales, no placeholder/untranslated strings.
- For shared brand/product terms that also appear in the mobile app ("pair", "streamer", "session", "Threadbase"), I match `tb-mobile`'s existing terminology choices per locale where the app already has an established rendering, for consistency between the app and the site.

## Metadata & SEO

- Each page's `metadata`/`generateMetadata` is localized (`title`, `description`) using the same message catalogs.
- Root layout emits `hreflang` alternate `<link>` tags for all 4 locale variants of each page, plus `x-default` pointing at the English URL.

## Out of scope

- No changes to `tb-mobile` or its locale files — read-only reference for terminology only.
- No visual/layout redesign beyond what RTL mirroring requires.
- No content changes to the English copy itself beyond what's needed to extract it into the message catalog (verbatim move).
